《高性能 MySQL》学习笔记（1）- 事务与隔离

Posted by Littlesqx on March 9, 2018

停滞了好久，想着找点东西写。

事务

事务是一组原子性的 SQL 查询，或者说一个独立的工作单元。如果数据库引擎能够成功地对数据库应用该组查询的全部语句，那么就执行该组查询；但如果其中有任何一条语句因为崩溃或其他原因无法执行，那么所有的语句都不会执行。简言之，事务内的语句，要么全部执行成功，要么全部执行失败（不执行）。

举一个例子，假设一个银行的数据库有张表：支票（checkings）表和储蓄（savings）表。现在要从用户 Jane 的支票账户转移 200 美元到她的储蓄账户，那么需要至少三个步骤：

  • 检查支票账户的余额是否不低于 200 美元。

  • 从支票账户余额钟减去 200 美元。

  • 在储蓄账户余额中增加 200 美元。

上述三个步骤的操作必须打包在一个事务中，任何一个步骤失败，都必须回滚所有步骤。

特征

1# 原子性（atomicity）

一个事务必须被视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚。

2# 一致性（consistency）

数据库总是从一个一致性的状态转换到另外一个一致性的状态。像上面三个步骤中，即便第二（三）个步骤失败了，但由于事务的回滚，使得最终数据库不会增加（减少）了 200 美元的数据。

3# 隔离性（isolation）

通常，一个事务所做的修改在最终提交前，对其他事务都是不可见的。但是不可见的隔离也产生了“副作用” ———— 不可重复读。后面会讨论不同等级的隔离级别。

4# 持久性（durability）

一旦事务提交，其所做的修改就会永久保存到数据库。

隔离级别

1# READ UNCOMMITTED（未提交读）

事务的修改，即使没有提交，对其他事务也都是可见的。事务可以读取未提交的数据，被称为“脏读”（Dirty Read）。这个级别会导致很多问题，从性能上来说，不会比其他级别好太多，但却缺乏其他级别很多好处，在实际应用中一般很少被使用。

2# READ COMMITTED（提交读）

一个事务从开始直到提交之前，所做的任何修改对其他事务都是不可见的。这个级别有时候也叫做不可重复读（nonrepeatable read），因为两次执行同样的查询，可能会得到不一样的结果。

3# REPEATABLE READ（可重复读）

解决了脏读的问题。该级别保证了在同一事务中多次读取同样记录的结果是一致的。但理论上，无法解决另一个幻读（Phantom Read）的问题，它指的是当某个事务在读取某个范围内的记录时，另外一个事务又在该范围内插入了新的记录，当之前的事务再次读取该范围内的记录时，会产生幻行（Phantom Row）。

MySQL 的默认事务隔离级别。

4# SERIALIZABLE（可串行化）

最高的隔离级别。它通过强制事务串行执行，避免了前面说的幻读问题。简单来说，SERIALIZABLE 会在读取的每一行数据上都加锁，所以可能导致大量的超时和锁争用的问题。

异常读

MySQL 的 @@global.tx_isolation 和 @@tx_isolation 配置项（分别对于全局和当前会话）设置隔离级别。

设置全局：

set global transaction isolation level [READ UNCOMMITTED|READ COMMITTED|REPEATABLE|SERIALIZABLE];

设置当前会话：

set session transaction isolation level [READ UNCOMMITTED|READ COMMITTED|REPEATABLE|SERIALIZABLE];
